Implementation Plan
Step 1 — Project Setup

Supabase project → run schema SQL
ChromaDB local instance (persistent mode, saved to disk)
FastAPI backend, React frontend
Seed org_knowledge table with 20–30 common orgs (HEC, Fulbright, DAAD, Google, etc.) and embed them into ChromaDB org_knowledge collection immediately


Step 2 — Student Profile

Build form → POST to /profile → insert into students
Narrate the profile into a sentence → embed → upsert into ChromaDB student_profiles
If Gmail: trigger OAuth flow → store gmail_refresh_token in students


Step 3 — Email Ingestion
Gmail path:

Use refresh token → get access token → call Gmail API with keyword filter
Fetch message bodies → decode base64 → insert into raw_emails
Log in gmail_sync_log

Manual path:

User pastes/uploads → insert directly into raw_emails


Step 4 — Cleaning Pipeline

For each unprocessed raw_email: strip HTML, signatures, footers
Deduplicate against existing raw_emails (same gmail_message_id or same sender+subject hash)
Save cleaned_body back to raw_emails


Step 5 — Classification

Embed cleaned_body → query ChromaDB opportunity_chunks for top-5 similar (cold start: skip this, use rules only)
Run rule-based signals (deadline keywords, opp keywords, CTA present)
Combine rule score + semantic similarity → classify
Insert row into opportunities with is_opportunity and confidence
Discard noise (set status = 'noise', stop processing)


Step 6 — Field Extraction

LLM call per opportunity with prompt: "Extract these fields from the email as JSON: opp_type, org_name, deadline, eligibility..."
Parse JSON response → update opportunities row with all extracted fields
Check deadline vs today → if past, set status = 'expired'


Step 7 — Org Knowledge Lookup

Take org_name from extracted fields → query ChromaDB org_knowledge collection
Retrieve top-1 match → use prestige_score and known_funding_type to fill gaps or validate extracted fields
If funding_type not in email but org is known as fully funded → fill it in, flag as inferred


Step 8 — Eligibility Hard Filter

Compare extracted cgpa_required, degree_required against student profile
Mark status = 'ineligible' if hard mismatch
Stop processing ineligible ones


Step 9 — Embedding Opportunities

For each active opportunity, chunk it into 5 chunks (header, eligibility, benefits, documents, deadline)
Embed each chunk → insert into ChromaDB opportunity_chunks with metadata


Step 10 — Semantic Scoring

Fetch student profile vector from ChromaDB student_profiles
Query opportunity_chunks filtered by student_id → get similarity scores per chunk
Average across chunks of same opportunity → semantic_score
Update opportunities.semantic_score


Step 11 — Deterministic Scoring

Calculate urgency_score from deadline days
Calculate profile_fit_score from field-by-field comparison (CGPA, degree, skills overlap, preferred types)
Calculate value_score from geo_scope, funding_type, prestige_score
Blend: final_score = (fit × 0.40) + (urgency × 0.35) + (value × 0.25) then mix with semantic: final = (deterministic × 0.6) + (semantic × 0.4)
Update opportunities with all scores


Step 12 — Explanation + Checklist Generation

For each top-N opportunity, LLM call with: extracted fields + scores + org context retrieved from ChromaDB
LLM outputs: explanation, action_checklist[], urgency_flag
Update opportunities row


Step 13 — API Endpoints (what frontend needs)
POST /auth/gmail          → OAuth flow
POST /profile             → create/update student
POST /emails/sync         → trigger Gmail fetch
POST /emails/manual       → paste emails
GET  /opportunities/{student_id}  → ranked list, ordered by final_score DESC
GET  /opportunities/{id}/detail   → full detail + checklist

Step 14 — Frontend

Profile form (Step 1 in UX)
Gmail connect button or manual paste box
Ranked opportunity cards with urgency color, score, explanation
Expandable card shows checklist + extracted fields
Filter bar: by type, by urgency, show ineligible/expired toggle


Data Flow Summary
Gmail / Paste
  → raw_emails (Supabase)
  → clean → classify (rules + ChromaDB similarity)
  → extract fields (LLM) → opportunities (Supabase)
  → org lookup (ChromaDB org_knowledge)
  → eligibility filter
  → embed chunks → ChromaDB opportunity_chunks
  → semantic score (ChromaDB profile vs chunks)
  → deterministic score (pure math)
  → blend scores → explanation (LLM)
  → final ranked list → frontendSonnet 4.6Adaptive
